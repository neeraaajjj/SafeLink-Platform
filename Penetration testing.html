Penetration Testing Roadmap

The standard methodology for conducting a professional penetration test, from scoping to reporting.

Phase 1: Pre-Engagement & Scoping

This is the most important phase for legal and professional success. Before you touch any systems, you must define the "rules of the game."

Key Activities

  • Define Scope: What is in scope (e.g., `*.domain.com`, `10.10.1.0/24`)? What is explicitly out of scope (e.g., third-party services, production servers)?
  • Set Objectives: What is the goal? To gain domain admin? To access a specific database? To test incident response?
  • Establish Rules of Engagement (ROE): When can testing occur (e.g., business hours only, or 24/7)? What techniques are forbidden (e.g., DoS attacks, social engineering)?
  • Legal & Contracts: Get a signed "Get Out of Jail Free" card. This is a contract (e.g., a Statement of Work) that legally authorizes your testing.

Phase 2: Reconnaissance (Information Gathering)

You can't attack what you don't know. The goal is to build a complete map of the target's attack surface.

Techniques & Tools

  • Passive Recon (OSINT): No direct contact with the target. Use Google Dorking, Shodan, Maltego, GitHub, LinkedIn, and public records to find subdomains, technologies, and employee names.
  • Active Recon: Directly probing the target. Use Nmap for port scanning, Amass or subfinder for subdomain enumeration, and DNS enumeration (dig, nslookup).

Phase 3: Scanning & Enumeration

You've found the doors; now you're jiggling the handles and peering in the windows. This phase involves a deep, active probe of the services you discovered.

Techniques & Tools

  • Vulnerability Scanning: Use tools like Nessus or OpenVAS to find "low-hanging fruit" (known vulnerabilities, missing patches).
  • Web Scanning: Use Nikto for web server misconfigurations and Dirb/Gobuster/ffuf to discover hidden files and directories.
  • Service Enumeration: Manually connect to services to get version info. For example, enumerate SMB (`enum4linux`), Active Directory (`BloodHound`), or SNMP. This is where you find the *exact* software version to search for exploits.

Phase 4: Gaining Access (Exploitation)

This is the "hack." You leverage a vulnerability found in the previous phase to gain an initial foothold (e.g., a shell) on a system.

Methods & Tools

  • Manual Exploitation: Using Burp Suite to perform SQL Injection, XSS, or Insecure Direct Object Reference (IDOR) attacks on a web app.
  • Automated Exploitation: Using the Metasploit Framework to launch a known public exploit (e.g., `MS17-010 EternalBlue`) against a vulnerable service.
  • Credential Attacks: Using Hydra or Medusa to brute-force a weak password on an SSH, FTP, or web login page.
  • Phishing: Using a tool like GoPhish (if in scope) to trick a user into giving up credentials or running a payload.

Phase 5: Post-Exploitation

You're in, but you're not done. The initial shell is often low-privilege. The goal now is to escalate privileges, move laterally to other systems, and achieve the objective.

Activities & Tools

  • Privilege Escalation: Find misconfigurations or kernel exploits to become `root` or `NT AUTHORITY\SYSTEM`. Tools: `linpeas.sh` (Linux), `winPEAS.exe` (Windows).
  • Lateral Movement: Use the compromised machine to attack the internal network. Pass-the-Hash (`Mimikatz`) in Windows/AD environments is common.
  • Persistence: (Optional, depends on scope) Establish a "backdoor" to maintain access, such as a cron job, a new admin user, or a reverse shell.
  • Data Exfiltration: Find and (safely) exfiltrate the "crown jewels" or "flag" defined in the objectives.

Phase 6: Reporting & Remediation

This is what the client pays for. A 100-page report is the final deliverable. It must be clear, professional, and provide actionable value.

Report Structure

  • Executive Summary: For non-technical managers. Summarizes the risk posture (e.g., "Critical") and business impact in simple terms.
  • Technical Summary: A narrative of the attack, from reconnaissance to post-exploitation, for the technical (Blue) team.
  • Findings & Vulnerabilities: One section for *each* vulnerability, detailing its risk (Critical/High/Med/Low), a description, and evidence (screenshots, code).
  • Remediation Steps: Clear, step-by-step instructions on how to *fix* each vulnerability.
